* 文書版数: 0.1
* 作成日: 2026-04-09
* 形式: Markdown
* 想定利用者: 個人開発者 / 小規模チーム
* 目的: レガシーシステムのソースコード解析から、保守運用に必要な各種設計書を自動生成する Web アプリケーションの実装仕様を定義する

\---

## 1\. 背景と目的

既存のレガシープログラムについて、設計書の紛失、開発担当者の退職、機能把握困難により、保守運用の継続が危機的な状態にある。  
本システムは、複数ソースファイルから成る既存プロジェクトを解析し、構造・処理・画面・DB更新の観点で設計情報を抽出し、Markdown および PlantUML ベースの図表として整理・生成することを目的とする。

本システムは、Claude Code のような対話型利用感を持ちつつ、内部的には **ワークフロー型 + ReAct 型のハイブリッド構成を採用したマルチエージェント** とする。

\---

## 2\. システムの到達目標

本システムは、アップロードされたソフトウェアプロジェクトに対して以下を実現する。

1. ソースコード、設定ファイル、SQL 定義、テンプレート、画面定義などを横断解析する
2. 言語やフレームワークを推定する
3. クラス・関数・画面・DB 更新箇所・処理フローなどを抽出する
4. 不完全情報がある場合でも、推定根拠付きで設計情報を構造化する
5. ユーザがチャットで補足指示を出しながら、再解析や特定範囲の深掘りができる
6. プロジェクトごとに Input / 中間成果物 / Output を分離管理する
7. 生成された設計書や図を Web UI 上で閲覧できる

## 5\. 基本方針

### 5.1 アーキテクチャ方針

本システムは以下の思想で設計する。

* 外側は **ワークフロー型**
* 各解析フェーズ内部は **ReAct 型**
* 役割別に **マルチエージェント** を分離
* 解析結果はすべて中間構造に正規化してから文書生成する

### 5.2 採用理由

ワークフロー型だけでは柔軟な探索が弱く、ReAct 型だけでは解析の暴走や粒度不整合が起こりやすい。  
そのため、フェーズ制御・状態管理・成果物管理はワークフローで行い、個別の探索・再読込・再推論は ReAct 的に実行する構成とする。

## 6\. 想定ユースケース

### 6.1 プロジェクト解析

1. ユーザが Web UI でプロジェクト一式をアップロード
2. システムがプロジェクトを登録
3. 自動で前処理・構成解析・言語判定・フレームワーク推定を実施
4. エージェント群が段階的に設計情報を抽出
5. 中間成果物を蓄積
6. 最終的に設計書群を生成
7. ユーザが設計書を Web 上で閲覧

### 6.2 チャットによる再解析

1. ユーザが「このクラスの責務を詳しく」などの指示を送信
2. 関連成果物と元コードを再読込
3. 必要に応じて部分再解析
4. 設計書を更新または追加生成

### 6.3 複数プロジェクト管理

1. ユーザが複数の案件を個別登録
2. プロジェクトごとに Input / 中間成果物 / Output を保持
3. UI 上で案件を切り替えてチャットと成果物を参照

## 8\. マルチエージェント構成

### 8.1 エージェント一覧

#### 8.1.1 Ingestion Agent

役割:

* アップロードファイルの展開
* 文字コード判定
* ディレクトリ構造収集
* 除外対象判定
* プロジェクトメタ情報作成

入力:

* アップロードファイル一式

出力:

* ファイル一覧
* ディレクトリ構造
* 前処理済み入力情報

#### 8.1.2 Language / Framework Detection Agent

役割:

* 言語判定
* ビルドツール判定
* フレームワーク推定
* DB 利用方式推定
* UI 種別推定

出力:

* 技術スタック推定結果
* 根拠一覧

#### 8.1.3 Code Structure Analysis Agent

役割:

* クラス / 構造体 / 関数 / メソッドの抽出
* 継承 / 実装 / 呼び出し関係の抽出
* モジュール責務推定

出力:

* クラス一覧
* メソッド一覧
* クラス図用中間データ

#### 8.1.4 Behavior Analysis Agent

役割:

* 処理フロー解析
* メソッド単位の処理概要抽出
* 呼び出し系列からシーケンス抽出
* フローチャート生成用中間情報作成

出力:

* 全体処理概要
* メソッド処理概要
* シーケンス候補
* フローチャート候補

#### 8.1.5 UI Analysis Agent

役割:

* 画面定義、テンプレート、イベント、遷移を解析
* 画面項目、イベント、レイアウトを推定
* GUI / Web / 独自画面の構成を整理

出力:

* 画面遷移図情報
* 画面項目仕様
* 画面イベント一覧
* 画面レイアウト情報

#### 8.1.6 DB Analysis Agent

役割:

* SQL、ORM、Mapper、Repository を解析
* CRUD の対応関係を整理
* DB 更新トリガとなる処理を抽出

出力:

* CRUD表
* DB更新仕様
* テーブル操作一覧

#### 8.1.7 Document Synthesis Agent

役割:

* 中間成果物を Markdown 文書へ整形
* PlantUML を生成
* 図と文章の相互整合を取る

出力:

* 各種 `.md`
* 各種 `.puml`

#### 8.1.8 Review / Gap Detection Agent

役割:

* 設計情報の不足箇所検出
* 矛盾検出
* 推定の信頼度低い箇所の再解析依頼
* ユーザへの確認質問候補作成

出力:

* 不足箇所一覧
* 再解析タスク
* 根拠不足警告

## 9\. ハイブリッド型の制御方式

### 9.1 ワークフロー型で制御する内容

* プロジェクト登録
* 解析ジョブ開始
* フェーズ遷移
* 再解析管理
* 途中成果物保存
* 最終出力生成
* UI への状態反映

### 9.2 ReAct 型で実行する内容

各エージェント内部で以下のループを許可する。

1. 状況理解
2. 次に読むべきファイル選定
3. 必要な追加解析
4. 解析結果の更新
5. 不足時の再探索

### 9.3 終了条件

各エージェントは以下で終了とする。

* 最大ステップ数到達
* 必要成果物が揃った
* 追加探索で新情報が増えない
* 上位オーケストレータが打ち切り判断した

### 10.2 モデル抽象化

モデル差異を吸収するため、以下の共通インターフェースを実装する。

```ts
interface ModelAdapter {
  generate(input: GenerateInput): Promise<GenerateOutput>
  toolCall(input: ToolCallInput): Promise<ToolCallOutput>
  stream(input: StreamInput): AsyncIterable<string>
  supportsJsonSchema(): boolean
  supportsToolCalling(): boolean
}
```

### 10.3 使い分け方針

* 計画作成、統合判断、レビュー: Azure OpenAI 優先
* ローカル解析、部分再生成、大量読み込み: Ollama も選択可能
* 実装上は設定で切替可能にする

## 12\. 非機能要件

### 12.1 前提

個人利用または小規模チーム利用を想定し、厳格なセキュリティ、監査、権限制御は本仕様では対象外とする。

### 12.2 性能

* 中規模プロジェクトで段階的解析が可能であること
* 長時間処理はジョブ形式で管理すること
* 再解析時は差分対象中心に処理すること

### 12.3 保守性

* エージェント追加が容易であること
* 言語対応を増やしやすい構造であること
* 文書テンプレート差し替えが容易であること

### 12.4 可観測性

* ジョブ状態を UI で確認できること
* 解析ログを段階的に参照できること
* エラー箇所が識別できること

## 23\. 制約事項

* 画面遷移や DB 更新は推定になる場合がある
* 信頼度と根拠の保持が必須

## 25\. 実装上の補足指針

* まずは中間構造の設計を優先する
* 文書生成は中間構造からの一方向生成にする
* エージェント同士で直接ファイル編集させず、オーケストレータ経由にする
* 解析結果は常に evidence を紐付ける
* ユーザ補足情報はプロジェクトメタ情報として保持する
* UI では「確定情報」と「推定情報」を分けて表示する

## 26\. まとめ

本システムは、レガシーな C / C++ / Java プロジェクトを対象に、失われた設計書を復元するための **ハイブリッド型マルチエージェント Web アプリ** である。  
Web UI によるプロジェクト単位の管理、チャットによる追加入力、Input / 中間成果物 / Output の分離管理、Markdown / PlantUML / SVG による設計書生成を中核とする。

実装の中心は以下である。

* ワークフロー型で全体制御
* ReAct 型で探索解析
* マルチエージェントで責務分離
* モデル切替可能な抽象化
* 中間成果物中心の設計
* プロジェクトごとの継続的な再解析

以上を本仕様の初版とする。
