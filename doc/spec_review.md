## 0. 総合評価

> [!IMPORTANT]
> 全体としてアーキテクチャ方針は堅実であり、ハイブリッド型マルチエージェント構成はこの種のタスクに適している。しかし、**「構文解析でできること」と「設計書に必要なこと」の間にある大きなギャップ**が仕様上で十分に埋められていない。以下のレビューは、そのギャップに焦点を当てる。

### 評価サマリ

| 観点 | 評価 | コメント |
|------|------|----------|
| アーキテクチャ方針 | ✅ 良好 | ワークフロー + ReAct のハイブリッドは適切 |
| データ管理設計 | ✅ 良好 | プロジェクト単位の分離、中間成果物の構造は実用的 |
| エージェント分割 | ⚠️ 要補強 | 役割は明確だが、エージェント間の情報連携仕様が不足 |
| 解析精度の設計 | ❌ 要対策 | 構文解析だけでは設計書品質にならない核心部分が未整備 |
| LLM 活用設計 | ⚠️ 要補強 | プロンプト戦略・コンテキスト管理が未定義 |
| 出力品質 | ⚠️ 要補強 | 人間エンジニアとの品質ギャップへの対策が不足 |

## 1. 核心的な懸念: 「構文解析」と「設計書作成」のギャップ

### 1.1 人間エンジニアは何をしているか

エンジニアがレガシーコードから設計書を書く時、実際にはこういうことをしている：

```
1. まずビルドして動かす（動作理解）
2. エントリーポイントから処理を追う（動的な理解）
3. 変数の値がどう変わるか脳内シミュレーションする
4. 「なぜこう書いたか」を推測する（意図の理解）
5. ビジネスロジックとコードの対応を見つける
6. 関連するドキュメント・DB定義・画面を横断して見る
7. 不明点はコードを何度も読み返す
8. 抽象度を上げて「何をしているか」を言語化する
```

### 1.2 Tree-sitter だけでできること

Tree-sitter は**構文解析器**であり、**意味解析器**ではない。

| 情報 | Tree-sitter でできるか | 設計書に必要か |
|------|------|------|
| クラス名・メソッド名の一覧 | ✅ できる | ✅ |
| 引数・戻り値の型 | ✅ できる | ✅ |
| 継承・実装の宣言 | ✅ できる | ✅ |
| import / include の宣言 | ✅ できる | ✅ |
| **ファイル横断の型解決** | ❌ できない | ✅ 必須 |
| **メソッド呼出し先のクラス特定** | ❌ できない | ✅ 必須 |
| **ポリモーフィズムの解決** | ❌ できない | ✅ 重要 |
| **マクロ展開後のコード** | ❌ できない | ✅ C/C++で必須 |
| **処理の意味・目的** | ❌ できない | ✅ 最重要 |

> [!CAUTION]
> **クラス図を正確に描くには、ファイル横断の型解決が不可欠。** 例えば `private UserService service;` の `UserService` がどのクラスか、どのパッケージにあるかを解決しないと、依存関係は描けない。Tree-sitter 単体ではこれができない。

### 1.3 提案: コード解析のレイヤを追加する

仕様に以下の解析レイヤを追加すべき：

```
Layer 1: 構文解析（Tree-sitter）→ AST、シンボル一覧
Layer 2: シンボル解決（自前ロジック）→ import/include の追跡、型の解決
Layer 3: 意味推論（LLM）→ 処理の意図、ビジネスロジック推定
```

**具体的に不足している仕組み:**

1. **シンボルテーブルの構築**
   - ファイル横断のシンボル（クラス、関数、変数）の定義・参照マッピング
   - Java: package + import から完全修飾名を構築する
   - C/C++: `#include` チェーンを辿ってヘッダの定義を集約する

2. **呼出しグラフの精度向上**
   - `obj.method()` の `obj` の型を解決しないと、呼出先クラスが特定できない
   - インターフェース経由の呼出しは実装クラスの候補を列挙する必要がある
   - C の関数ポインタ経由の呼出しは静的解析の限界を明示すべき

3. **C/C++ プリプロセス対応**
   - `#define`, `#ifdef` によるコード分岐
   - マクロ展開なしに正確な解析は不可能
   - 最低でも `gcc -E` or `clang -E` 相当のプリプロセス結果を解析対象にする選択肢を用意すべき

## 2. LLM 活用の設計が不十分

### 2.1 プロンプト戦略が未定義

仕様書にはプロンプトの設計方針が一切記載されていない。しかし **LLM の出力品質はプロンプト品質に直結する**。

**追加すべき仕様:**

- エージェントごとのプロンプトテンプレート仕様
- 出力スキーマの明示（JSON Schema で構造化出力を強制する）
- Few-shot 例の管理方針
- 言語（日本語 / 英語）の出力制御方針

```
// 例: クラス責務推定のプロンプト設計がないと…
// こういう曖昧な依頼しかできない：
"このクラスの責務を教えてください"

// 本来はこうあるべき：
"以下のクラスのコードを分析し、JSON形式で出力してください:
{
  "className": "...",
  "responsibility": "1-2文で責務を記述",
  "layer": "Controller|Service|Repository|Entity|Utility|Other",
  "relatedDomainConcepts": ["..."],
  "confidence": 0.0-1.0
}

[クラスのソースコード]
[関連するimport/依存クラスの情報]"
```

### 2.2 コンテキスト管理戦略がない

大規模プロジェクトでは、全ファイルをLLMに渡すことは不可能。しかし **部分的な情報だけでは設計書の品質が著しく劣化する**。

**追加すべき仕様:**

| 項目 | 必要な定義 |
|------|-----------|
| チャンキング方針 | ファイル単位 / クラス単位 / メソッド単位のどれか |
| コンテキスト予算 | エージェントごとのトークン予算配分 |
| 要約チェーン | 長いファイルの段階的要約方法 |
| 関連コード選択 | 解析対象メソッドに対して、呼出元/呼出先のコードをどこまで含めるか |
| 累積コンテキスト | ReAct ループ内で過去のステップの結果をどう保持するか |

> [!WARNING]
> **コンテキスト管理を設計しないと、以下が起こる:**
> - 100ファイルのプロジェクトで「全体処理概要」を書こうとしても、LLM は全体を見れない
> - 呼出チェーンを追跡しようとしても、呼出先のコードがコンテキストにない
> - 結果的に、各ファイルの断片的な要約を並べただけの「全体概要」になる

### 2.3 提案: 「段階的コンテキスト構築」戦略

```
Step 1: 全ファイルの「1行サマリ」を作る（ファイルパス + 責務推定1行）
Step 2: サマリ一覧をLLMに渡し、全体アーキテクチャを推定する
Step 3: アーキテクチャに基づいて重要ファイルを選定する
Step 4: 重要ファイルの詳細解析を行う
Step 5: 詳細解析結果 + 全体サマリから設計書を生成する
```

この戦略を §9「ハイブリッド型の制御方式」に明記すべき。

## 3. 中間データスキーマの具体性不足

### 3.1 問題点

spec.md §13.3 で中間成果物として「AST風構造」「シンボルテーブル」「呼び出しグラフ」等を列挙しているが、**具体的なデータスキーマがない**。

エージェント間のデータ連携はこの中間データを介して行われるため、スキーマの曖昧さはシステムの根幹に影響する。

### 3.2 提案: 主要中間データのスキーマを定義する

最低限、以下を spec.md に追加すべき：

**クラス情報の中間スキーマ（例）:**

```json
{
  "id": "cls_001",
  "name": "UserService",
  "fqn": "com.example.service.UserService",
  "filePath": "src/main/java/com/example/service/UserService.java",
  "language": "java",
  "type": "class",
  "modifiers": ["public"],
  "superClass": { "name": "BaseService", "fqn": "com.example.base.BaseService" },
  "interfaces": [{ "name": "IUserService", "fqn": "com.example.api.IUserService" }],
  "fields": [
    { "name": "userRepository", "type": "UserRepository", "modifiers": ["private"] }
  ],
  "methods": ["mtd_001", "mtd_002"],
  "layer": "Service",
  "responsibility": "ユーザの登録・更新・削除を管理する",
  "confidence": 0.85,
  "evidences": ["ev_001", "ev_002"]
}
```

**呼出関係の中間スキーマ（例）:**

```json
{
  "callerId": "mtd_001",
  "calleeId": "mtd_010",
  "callType": "direct|interface|virtual|function_pointer",
  "lineNumber": 42,
  "isConditional": true,
  "confidence": 0.7
}
```

## 4. 出力ドキュメントの品質に関する懸念

### 4.1 「メソッド処理概要」の実現可能性

spec.md §15.6 の「メソッド処理概要」は、人間エンジニアが書く場合、以下を含む:

```markdown
## transferMoney(fromAccount, toAccount, amount)

### 前提条件
- fromAccountが有効であること
- toAccountが有効であること
- amountが正の数であること

### 主処理
1. fromAccountの残高を確認する
2. 残高が不足している場合、InsufficientFundsExceptionをスローする
3. fromAccountから金額を減算する
4. toAccountに金額を加算する
5. トランザクションログを記録する

### 例外系
- InsufficientFundsException: 残高不足時
- AccountNotFoundException: 口座が存在しない場合
```

**問題:** これを自動生成するには、コードの**意味**を理解する必要がある。Tree-sitter で extract できるのは `if (balance < amount) throw new ...` という構文であり、「残高不足チェック」という意味は LLM に推論させる必要がある。

**リスク:** LLM がハルシネーションで **コードにない処理を書いてしまう** 可能性がある。

**提案:** 以下のガードレールを仕様に追加すべき：

- メソッド処理概要は、必ず **ソースコードの行番号参照** を付与する
- LLM の推論結果には `[推定]` マーカーを付与し、コードから直接読み取れる情報と区別する
- 生成結果と元コードの **整合性チェック**（呼出先メソッドが実在するか、例外クラスが実在するか）を行う

### 4.2 「シーケンス図」の実現可能性

シーケンス図の生成はこのシステムで**最も困難な機能**の一つ。

| 課題 | 理由 |
|------|------|
| 「代表ユースケース」の選定 | コードからユースケースを推定する必要がある。LLM でも精度が低い |
| 呼出チェーンの追跡 | ポリモーフィズム、DI、イベント駆動では静的解析が困難 |
| 非同期処理のモデリング | スレッド、コールバック、Future/Promise の追跡は極めて難しい |
| 適切な抽象度の選択 | 全呼出を描くと巨大になり、省略しすぎると意味がない |

**提案:**

- Phase 1 ではシーケンス図を **対象外** とし、Phase 2 に移動（現在の spec.md と task_plan.md は既にこうなっている ✅）
- シーケンス図は **ユーザが起点メソッドを指定する** 対話型にする
- 深さの制限（呼出し追跡の最大深度）を設定できるようにする
- 自動選定の場合は、Controller のパブリックメソッドを起点候補にする

### 4.3 「CRUD 表」の精度問題

CRUD 表は比較的実現しやすいが、以下の落とし穴がある：

- MyBatis の動的 SQL（`<if>`, `<choose>`）内の操作は条件付き
- JPA の `save()` は INSERT か UPDATE か判定できない（`merge` の挙動）
- Repository パターンで `findBy...` メソッド名規約から推定する場合の精度
- ネイティブ SQL とメソッド名の不一致

**提案:** CRUD 表に **「確定」と「推定」のマーキング** を入れる仕組みを仕様化する

## 5. 運用・実用面の懸念

### 5.1 解析時間の見積もりがない

中規模プロジェクト（500ファイル、10万行程度）を解析した場合に、どの程度の時間がかかるか？

- Tree-sitter 解析: 数秒〜数十秒（高速）
- LLM による推定: **数分〜数十分**（ファイル数 × API呼出し）
- PlantUML → SVG 変換: 数秒

**LLM 呼出しがボトルネック** になることは確実。仕様に以下を追加すべき：

- 推定処理時間の目安
- LLM 呼出し回数の最適化方針（バッチ化、キャッシュ）
- ユーザへの所要時間通知

### 5.2 コスト管理

Azure OpenAI を使う場合、大規模プロジェクトの解析は **かなりのトークン消費** になる。

- 500ファイル × 平均200行 = 10万行 ≈ 400万トークン（入力のみ）
- 各ファイルをLLMに複数回渡す（責務推定、処理概要、等）
- 合計: **数百万〜数千万トークン** の可能性あり

**提案:** トークン消費量の見積もり表示、コスト上限設定を仕様に含めるべき

### 5.3 増分解析の仕組みがない（Phase 1）

Phase 1 では、ファイルを 1 つ変更しても全解析をやり直す設計になっている。Phase 3 に差分再解析があるが、**MVP 段階でも最低限のキャッシュ** は必要：

- Tree-sitter 解析結果のファイル単位キャッシュ（ファイルハッシュベース）
- LLM 推論結果のキャッシュ（同一入力に対するキャッシュ）

## 6. 見落とされている仕様項目

### 6.1 エラーリカバリと部分成功

仕様に **一部のファイル解析が失敗しても全体として結果を出力する** 仕組みの記述がない。

レガシーコードでは:
- 文法が古くて Tree-sitter がパースできないファイルがある
- 文字コードが不正で読めないファイルがある
- 巨大ファイル（数万行）で LLM のコンテキスト制限を超える

**提案:**
- 各ファイルの解析結果に `status: success|partial|failed` を付与
- 失敗ファイル一覧を「解析レポート」として出力
- 失敗ファイルがあっても、成功分で設計書を生成する

### 6.2 ユーザによるフィードバック・修正の仕組み

生成された設計書に対して、ユーザが:
- 「この責務推定は違う」と修正する
- 「このクラスはこう使われている」と補足する
- 「このシーケンスは正しい」と承認する

こうした **ユーザフィードバックを次回解析に反映する仕組み** が仕様にない。
チャット再解析（§6.2）はあるが、**構造化されたフィードバック**（承認/修正）の保持方法が未定義。

### 6.3 生成物間の一貫性保証

別々のエージェントが生成する以下の文書は、相互に一貫している必要がある：

- クラス一覧に書かれたクラスが、クラス図にも存在する
- メソッド一覧のメソッドが、CRUD 表にも反映される
- 全体処理概要に書かれた流れが、シーケンス図と矛盾しない

**現在の仕様には、この一貫性を保証する仕組みがない。**

**提案:** Document Synthesis Agent は、生成後に以下のクロスチェックを行うべき：
- クラス一覧のIDセット == クラス図のノードセット
- メソッド一覧のIDセット ⊇ CRUD表で参照されるメソッドセット
- 不整合があれば警告として出力する

## 9. まとめ

この仕様書は、レガシーコード解析のための Web アプリケーションとして**骨格は十分**に定義されている。しかし、**人間エンジニアと同等レベルの設計書**を自動生成するには、以下の3つの根本的なギャップを埋める必要がある：

1. **構文解析 → 意味解析のブリッジ**: Tree-sitter だけでは「構造」は取れるが「意味」は取れない。ファイル横断のシンボル解決レイヤと LLM による意味推論の組み合わせ戦略が必要。

2. **LLM の戦略的活用**: プロンプト設計、コンテキスト管理、段階的要約の方針なしに LLM を使うと、出力品質が不安定になる。

3. **生成物の信頼性保証**: Evidence / confidence の概念は導入されているが、具体的なスコアリング方法、生成物間の整合性チェック、推定と確定の区分が未定義。

**これらが解決されれば、「完璧ではないが実用的な設計書の自動生成」は十分に実現可能** だと評価する。特に、構造的な情報（クラス一覧、メソッド一覧、CRUD表、クラス図）は高い精度が期待でき、意味的な情報（処理概要、責務推定）は LLM の品質に依存するものの、Evidence と confidence で信頼性を可視化できる設計になっている。
